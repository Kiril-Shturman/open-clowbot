## 2) Что ещё не доведено до финального состояния

### A. Не завершена полная Astro-миграция
В `public/` всё ещё лежат страницы/контент, которые остаются статики и не переведены в полноценные Astro-шаблоны:
- `public/account/index.html`
- `public/cabinet/index.html` (дубль/наследие)
- статьи в `public/guides/*.html`

Это значит, что сейчас у проекта всё ещё **два источника правды**:
- Astro pages в `src/pages`
- старые HTML в `public`

### B. Гайды не мигрированы как контент-система
Сейчас:
- index гайдов уже в Astro
- сами статьи остаются статическими html в `public/guides/`

Нужно перевести статьи в нормальную Astro-структуру:
- либо `src/pages/guides/*.astro`
- либо через content collections

### C. Кабинет пока выглядит как техничный MVP
`src/pages/cabinet/index.astro` работает как utilitarian dashboard:
- ручной ввод `userId`
- ручной ввод JWT
- простая загрузка wallet/subscriptions/servers/payments

То есть backend hooks есть, но UX пока не финальный продуктовый.

### D. Auth flow выглядит недожатым end-to-end
Есть страницы login/register и обращения к:
- `/api/auth/google`
- `/api/auth/login`
- `/api/auth/register`

Но по текущему быстрому обзору в backend-роутах явно не видно полноценного `auth` route-файла в `src/routes/`.
Это надо отдельно проверить и либо подключить, либо дописать, либо синхронизировать frontend с фактическим API.

### E. Прод-готовность backend не подтверждена боевым прогоном
Есть сильный MVP, но до уверенного прод-статуса нужно подтвердить:
- реальные env
- реальная PostgreSQL
- применённые миграции
- рабочие webhook secrets
- фоновые worker-процессы
- end-to-end сценарии

## 3) Что прямо сейчас считаю готовым

### Готово
- Архитектурный старт Astro-проекта
- Основные страницы сайта
- Брендовая визуальная база
- Сборка проекта (`dist` уже есть)
- Backend skeleton/MVP с платёжкой, подписками, deploy и AI proxy

### Частично готово
- кабинет
- auth
- гайды
- интеграция фронта и backend
- production hardening

### Не готово
- единая чистая Astro-структура без наследия из `public`
- финальный UX личного кабинета
- подтверждённый боевой сценарий: регистрация → оплата → активация → деплой → кабинет
- чистка дублей и хвостов

## 4) Самые важные задачи сейчас

### Приоритет 1 — убрать двусмысленность структуры
Нужно зафиксировать правило:
- всё новое делаем только в `src/pages` / `src/components` / `src/styles`
- `public/` оставляем только для ассетов и временно для legacy-контента

### Приоритет 2 — добить контент миграцию
Перенести из legacy-статики:
- статьи `/guides/*.html`
- при необходимости `account`/старые дубли кабинета

### Приоритет 3 — проверить auth API
Сверить фронт и backend:
- существуют ли реально `/api/auth/login`
- `/api/auth/register`
- `/api/auth/google`
- если нет — доделать
- если есть в другом месте — привести к одному контракту

### Приоритет 4 — привести кабинет к продуктовой версии
Убрать ручной режим с `userId` и `JWT` в интерфейсе и перейти на нормальную пользовательскую сессию.

### Приоритет 5 — прогнать боевой чеклист
Минимальный обязательный прогон:
1. регистрация
2. логин
3. Google login
4. создание платежа
5. webhook
6. создание/обновление подписки
7. деплой на сервер
8. отображение в кабинете

## 5) Жёсткий вывод

Проект **не сырой**, а уже хорошо продвинутый.
Но сейчас он в стадии:
- **сильный foundation**
- **не завершённая миграция**
- **backend MVP есть, но нужен финальный прогон и дожим UX**

Если коротко:
- маркетинг/лендинг: **почти готово**
- Astro как основа: **принято и уже работает**
- backend: **почти собран, но ещё не подтверждён как полностью боевой**
- главная проблема: **дубли, legacy-хвосты и недожатые end-to-end сценарии**

## 6) Следующий разумный шаг

Следующий лучший ход — не косметика, а:

1. провести **структурную зачистку Astro-проекта**,
2. составить **точный migration map legacy → Astro**,
3. проверить **реальное наличие auth API**,
4. после этого уже добивать конкретный прод-сценарий.
